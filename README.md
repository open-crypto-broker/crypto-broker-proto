# Crypto Broker Protobuf

This repository contains the definitions for the protobuf messages that will be shared across the Crypto Broker clients and the server. This repository is imported as a Git submodule on each client and server in order to use the exact same version on all apps.

## Message structure

The definitions are split into an application-facing crypto service (`CryptoGrpc`), intended for client applications, and an internal development service (`CryptoGrpcDev`), used for benchmarking and testing.

### Shared messages

- **Metadata**:
  Contains basic information about a request, such as a unique ID. This helps track and identify requests. It optionally carries a `TraceContext`.

- **TraceContext**:
  Holds trace context for manual propagation across services, including the trace ID, span ID, trace flags, trace state and a correlation ID.

### Output formats

- **HashOutputFormat**:
  Selects the encoding of the returned hash value: `HEX` (default) or `RAW`.

- **SignOutputFormat**:
  Selects the encoding of the returned signed certificate: `DER` (default) or `PEM`.

### Application-facing service: `CryptoGrpc`

#### `CryptoGrpc` messages

- **HashDataRequest**:
  Used to ask the server to hash some input. It includes the Crypto Profile to use, the input to be hashed, the desired output format and a Metadata.

- **HashDataResponse**:
  Contains the resulting hash value (as hex or raw bytes), the used hash algorithm and Metadata about the hashing process.

- **SignCertificateRequest**:
  Used to request a digital signature for a certificate signing request (CSR). It includes the Crypto Profile, CSR data, signing key, validity period, CA certificate, the desired output format and Metadata. Optionally, a subject and CRL distribution point URLs can be added.

- **SignCertificateResponse**:
  Contains the signed certificate generated by the server (as PEM or DER) and the metadata.

#### `CryptoGrpc` functions

- **HashData**:
  Takes a `HashDataRequest` and returns a `HashDataResponse`. This is used to generate a hash of the provided input using the hashing algorithm specified by the profile.

- **SignCertificate**:
  Takes a `SignCertificateRequest` and returns a `SignCertificateResponse`. This is used to sign a CSR based on the profile's settings and return the signed certificate.

### Internal development service: `CryptoGrpcDev`

#### `CryptoGrpcDev` messages

- **BenchmarkRequest**:
  Request a performance benchmark from the server.

- **BenchmarkResponse**:
  The result of the Crypto Broker server performance benchmark.

- **FakeEndpointRequest**:
  Used to invoke a fake endpoint for testing and connectivity checks.

- **FakeEndpointResponse**:
  Contains a message returned by the fake endpoint along with the metadata.

#### `CryptoGrpcDev` functions

- **Benchmark**:
  Takes a `BenchmarkRequest` and return a `BenchmarkResponse`. This is used to get performance overview of the different cryptographic algorithms and functions.

- **FakeEndpoint**:
  Takes a `FakeEndpointRequest` and returns a `FakeEndpointResponse`. This is used for testing and connectivity checks.

## Protobuf as a submodule

To add this to your project as a submodule you can run this command from the repository where you want to add it. Please change the URL for this repository's URL and establish the subfolder where you want to save this inside your project.

```shell
git submodule add https://github.com/open-crypto-broker/crypto-broker-proto <sub/folder/protobuf>
```

To fetch the latest commit from this repository, you can just run:

```shell
git submodule update --init
```

This will fetch the latest changes. Moreover, VS Code provides already functionality to fetch the code if you go to the `Source Control` section of VS Code.
